🌐 全端實戰作品 Full-Stack E-commerce Project

  • 外語學習書城：全端商務管理系統 [核心功能完備 - Feature Complete]
    技術棧： Java (Spring Boot 3, Spring Security, JPA) / Vue 3 (Composition API, Pinia) / PostgreSQL / Docker
    1. 動態互動評論與數據關聯 (UGC System)： 實作用戶生成內容模組，利用 JPA 實體關聯管理 Book-User-Review 的複雜數據鏈條。結合 Spring Security 權限驗證，確保只有登入用戶可發布內容，達成完整的業務邏輯閉環。
    2. 多維度檢索與高擴展語種篩選： 實作 14 種學習語系下拉式篩選 與關鍵字（書名、作者、ISBN）複合搜尋。開發過程中成功處理 PostgreSQL 層級的 Check Constraint 約束擴充，展現系統在業務擴張時的底層調優能力。
    3. 嚴謹的異質支付狀態機與支付路徑閉環： 針對信用卡與 COD 設計異質 訂單狀態機 (Order State Machine)。實作 AWAITING_PAYMENT 原子中間狀態，防止未扣款即跳轉的邏輯漏洞；並開發「狀態感知支付回溯連結」，確保用戶在意外斷連後仍能穩定完成交易閉環。
    4. 原子性事務 (ACID) 與庫存自動補償邏輯： 深度應用 Spring @Transactional 管理事務。除了確保下單時的原子性以防止超賣，更實作了「訂單取消自動回補」機制：當訂單狀態轉為 CANCELLED 時，系統自動觸發庫存補償邏輯，確保數據最終一致性。
    5. 無狀態認證與法律合規性設計： 採用 JWT 實現前後端分離認證，並主動實作 Demo 模式警示機制，於支付環節標註「模擬交易」警語，體現對個人資料安全與法律規範的專業意識。
    6. 生產環境部署優化與 SPA 路由重定向： 獨立完成 Docker 容器化部署。針對 SPA 在生產環境下「頁面刷新導致 404」的典型衝突，透過配置 vercel.json 的 Rewrite 轉發規則 重新定義伺服器請求路徑，確保前端路由與伺服器環境的無縫銜接。

  1. 豎琴模擬器《Harp Real》分析與演奏網站 (3 頁)
    **技術棧：** Vue 3 / Vue Router / HTML / CSS / 原生 JavaScript
    • 現代前端框架實踐：
      將多個原始 HTML 頁面重構成單頁應用程式 (SPA)，實踐 Vue 3 的組件化開發思維，並利用 Vue Router 實現高效的頁面導航。
    • 工程化與外部服務整合：
      抽象化共享功能（如 Footer、社交分享、Google Translate API 初始化、Facebook SDK 載入）為獨立的 Vue 組件，利用 onMounted 生命週期鉤子確保外部腳本正確且不重複地載入。
    • 內容與技術分析：
      深入探討數位樂器應用程式的設計原理、音高限制以及實作技術推測，展現跨領域的分析能力。
    • 響應式設計：
      確保在各種設備上，網站內容和影片都能保持良好的可讀性與佈局。
    • 錨點導航處理：
      Vue 3 的 SPA 不會像傳統 HTML 那樣在換頁時重新載入整個 DOM，因此原生的 <a href="#section1"></a> 不會自動觸發瀏覽器的捲動行為。為了讓頁內導航正常運作，我改用 JavaScript 的 scrollIntoView() 並在路由切換後手動觸發捲動，實作出與傳統錨點相同的效果。 透過這點，我進一步理解了 SPA 與傳統多頁網站（MPA）在瀏覽器行為與滾動控制上的本質差異。
    • Vue SPA 部署：
      Vue 專案不能像純 HTML 一樣直接上傳，因此我使用 Vite 進行打包並部署至 GitHub Pages。由於靜態主機無法處理 Vue Router 的子路由，我採用 Hash History 避免刷新後出現 404，同時調整 base URL 與資源路徑，使所有頁面與靜態檔案在部署環境中都能正常載入。完成這個過程讓我熟悉了 Vue 的打包流程、Router 的部署差異，以及靜態主機的限制與解法。
    • Google Translate API 元件提示：
      由於本網站使用 Vue 3 SPA 架構，頁面切換時不會重新載入整個 DOM，可能導致 Google Translate 選單暫時無法顯示。若遇到此情況，請按 F5、Ctrl+R 或刷新頁面，即可正常顯示語言選單。
